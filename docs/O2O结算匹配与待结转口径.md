# O2O 三平台「结算金额」与「待财务匹配」口径

> 与 `sql/o2o_daily_store_aggregate.sql` 一致；展示板可读取该查询输出的 `*_pending_fin_match_total` 与门店级 `*_pending_fin_cnt`。

**数据源**：仅依赖 PostgreSQL 中 **`ods_rpa_*` 贴源表**（FTP → `ingest.py` 入库，非各平台 Open API）。文件与表名对照见 **`docs/FTP_TO_ODS_MAPPING.md`**。当前实现不依赖 dbt；只要映射表里有数据即可计算。

## 1. 问题背景

- **美团**：前台 **商品明细**（`ods_rpa_meituan_product_detail`）按 **下单时间** 落在业务日 D；**商家应收款** 只在 **财务订单明细**（`ods_rpa_meituan_fin_order`）出现，且账单常 **T+1 或更晚**。若只用「当日分区 dt = D 的财务文件」或按财务下单时间筛，会漏单或把未出账订单误算成 0。
- **京东**：订单（`ods_rpa_jd_order`）与 SKU 对账单（`ods_rpa_jd_finance`）分离；原先仅 `jf.dt = D` 时，账单晚到的订单会被直接丢掉而非显式待匹配。
- **饿了么**：订单导出 **「商户应收金额」** 与财务 **「订单应收」**（ODS：`"订单应收"`）为同一结算语义；为与美团/京东 **同一套「财务为准、未匹配则挂起」** 逻辑，增加 **订单锚点 + 销售账单明细** 匹配；未匹配订单计入待结转，**当日结算额只计已匹配部分**（金额取财务侧 **订单应收**，**非**净营业额合计）。字段对照见 **`docs/饿了么结算字段对照.md`**。

## 2. 通用规则

| 概念 | 定义 |
|------|------|
| **业务日 D** | `params.d`；时间窗半开区间 `[D, D+1)`，锚在 **下单时间**（美团商品明细 / 饿了么订单）或 **成交时间**（京东订单，缺则用门店收单时间）。 |
| **财务全表窗口** | 为控制扫描量，财务表限定 `dt >= D - 120 天`（与 `params` 同库内计算）。后续可改为物化视图或累计表。 |
| **已匹配** | 锚点订单号在对应财务表中能关联到至少一行（京东为 `到家业务单号` 汇总结算金额）。 |
| **待参与计算** | 锚点满足业务日 D 与非取消规则，但在财务窗口内 **仍无匹配** 的订单；**不计入当日结算金额**，只在「待匹配单数」中展示。 |
| **当日结算金额** | 仅 **已匹配** 订单：美团/京东取财务金额；饿了么取 `ods_rpa_eleme_fin_sales_detail` 窗内按订单 **`SUM("订单应收")`**（**不是**净营业额合计，**不是**只取一行）。 |

## 3. 平台细则

### 美团

- **锚点**：`ods_rpa_meituan_product_detail`，`订单状态` ≠ `已取消`，`下单时间 ∈ [D,D+1)`，`DISTINCT ON (订单编号)` 取 `订单完成时间` 最新一行；门店键 `商家ID` / `商家名称` 对齐 `dim_store_o2o`（`美团买药`）。
- **财务**：`ods_rpa_meituan_fin_order`，`基础信息_订单状态` ≠ `订单取消`，不按「业务日」截断文件；`DISTINCT ON (基础信息_订单号)` 取 `基础信息_账单日期` 最新，`商家应收款` 为结算额。
- **企客**：仅对 **已匹配** 单数 × `dim_store.meituan_enterprise_rebate`。

### 京东

- **锚点**：`ods_rpa_jd_order`，分区 `dt = D`（与 RPA 日文件夹一致），`成交时间/门店收单时间 ∈ [D,D+1)`，`订单状态` ≠ `已取消`（可按实际枚举扩展）。
- **财务**：`ods_rpa_jd_finance`，`dt >= D-120`，按 `到家业务单号` **SUM(结算金额)**。

### 饿了么

- **锚点**：沿用原 `ods_rpa_eleme_order` 去重剔除规则。
- **财务**：`ods_rpa_eleme_fin_sales_detail`，`dt >= D-120`，按 **`"订单号"` 汇总 `SUM("订单应收")`** 为结算额（一单多行须相加）。净营业额、结算金额列仅作参考，详见 **`docs/饿了么结算字段对照.md`**。

## 4. 展示板字段建议

| 字段 | 粒度 | 含义 |
|------|------|------|
| `mt_pending_fin_match_total` | 报表日一行（可每行重复） | 美团当日锚点未匹配财务的单数 |
| `ele_pending_fin_match_total` | 同上 | 饿了么 |
| `jd_pending_fin_match_total` | 同上 | 京东 |
| `mt_pending_fin_cnt` | 门店 | 该店美团待匹配单数 |
| `ele_pending_fin_cnt` | 门店 | 该店饿了么待匹配单数 |
| `jd_pending_fin_cnt` | 门店 | 该店京东待匹配单数 |

## 5. 双逻辑校验 SQL

- **`sql/o2o_eleme_order_fin_settlement_compare.sql`（推荐）**  
  **仅饿了么**：同日 RPA **订单明细** + 财务窗 **销售账单明细** 即可。**主对比（规则口径）**：已取消/全退时导出 **正→0、负保留**；否则取 **`商户应收金额`**；财务 **`SUM("订单应收")` 按订单号**。另输出诊断列与「取消/全退规则与财务仍不一致」笔数；`pending_fin_not_matched_cnt` = 锚点单无账单匹配。另保留分项式与参考列。

- **`sql/o2o_settlement_two_logic_validation.sql`**  
  三平台粗验（含美团展开式、饿了么、京东）；饿了么逻辑一为 **订单应收** 汇总，与上一文件及日聚合一致。

未匹配财务的订单数：主聚合里 `ele_pending_fin_cnt`；饿了么专用脚本里 `pending_fin_not_matched_cnt`。

## 6. 后续（费用拆解）

在结算锚点与匹配关系校准后，可在 **已匹配** 子集上展开美团/京东财务多列（佣金、配送服务费、补贴等）做「扣除项 / 平台承担 / 商家承担」明细；未匹配订单不参与该项分析。
