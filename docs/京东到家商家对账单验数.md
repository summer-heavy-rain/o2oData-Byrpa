## 1.1 美团 / 饿了么 / 京东 通用规则：订单匹配 + 公式校验（三平台铁律）

### 第一步：按订单号匹配

每期对账时，先将 **订单明细**（订单表）的订单号与 **财务账单**（财务表）的订单号做匹配：

| 匹配结果 | 处理 |
|---------|------|
| **能匹配到** → 进入第二步公式校验 | 该订单有财务数据，用公式推导 vs 账单结算金额比对 |
| **匹配不到**（订单有、财务账单无） | **暂存不纳入当期计算**。等财务账单更新后该订单出现了，再计算并算给门店。**但该部分收入必须单独列出**（按门店汇总），供 xiaohou 核查确认 |

**「暂存」输出格式**：

```
📋 未匹配财务账单的订单（暂存不计入当期）
| 门店 | 订单数 | 订单金额合计 | 最早下单时间 | 最晚下单时间 |
|------|--------|------------|------------|------------|
| XX店 |   12   |   ¥3,456   | 2026-03-25 | 2026-03-28 |
| ...  |  ...   |    ...     |    ...     |    ...     |
合计：XX 笔，¥XX,XXX — 待财务账单补齐后纳入
```

**关键**：暂存订单**不是丢掉**，是延后到财务账单覆盖后再计算。收入单独列出是为了让 xiaohou 知道当期有多少金额还未经财务确认，避免影响决策判断。

### 异常检测：到家订单有但财务账单整批缺失（三平台通用）

如果订单明细中 **某订单来源（如到家）存在订单**，且订单时间 **已经比较靠前**（如 ≥3 天前），但对应的 **财务账单一条都没有**（不是个别订单匹配不到，而是该来源/该平台整批为零），**基本可以确定是 RPA 没抓取到**。

AI 必须 **主动告知 xiaohou**：

```
⚠️ 到家财务账单疑似 RPA 缺采
订单来源「到家」有 XX 笔订单（最早 YYYY-MM-DD），
但财务账单中该来源 0 条记录。
基本判断：RPA 未抓取到家财务账单。
请确认影刀 RPA 对应脚本（R15 京东到家财务）运行状态。
```

此规则同样适用于其他平台/订单来源：**只要订单有、财务整批空、且时间已不是当天**，都按 RPA 缺采处理，通知 xiaohou 排查。

### 退款纳入 + 延迟退款归类（三平台通用）

退款类交易行（整单退款/部分退款/退款冲抵等）**必须纳入**结算计算，不能只取正向订单。

| 场景 | 处理 |
|------|------|
| 退款订单号 **在当日订单明细中** | 正常纳入当期结算，退款金额抵扣对应门店 |
| 退款订单号 **不在当日订单明细中**（仅财务账单有） | 说明是 **延迟退款**——客户之前下单，非当天退款导致财务账单滞后。**单独归类为「延迟退款金额」**，在门店结算中独立一行扣减，方便 xiaohou 核查 |

**「延迟退款」输出格式**：

```
📋 延迟退款（财务账单有退款行，当日订单明细无对应订单号）
| 门店 | 退款笔数 | 退款金额合计 | 退款类型 | 订单号样本 |
|------|---------|------------|---------|-----------|
| XX店 |    3    |   -¥326.66 | 订单退款 | 270205... |
| ...  |   ...   |    ...     |   ...   |    ...    |
合计：XX 笔，-¥XX,XXX — 归入延迟退款扣减
```

**关键**：延迟退款**必须算给门店**（扣减），只是单独列出来让 xiaohou 知道哪些是滞后的退款、金额多大，避免跟当日经营数据混淆。

### 第二步：公式推导 vs 账单结算金额（仅对已匹配订单）

对第一步中 **能匹配到** 的订单，**必须用已有的结算公式（各分项代数和）算一遍推导值**，再与财务账单中的 **平台结算金额**（京东=`应结金额` / 饿了么=`结算金额` / 美团=`商家应收款`）**逐单比对**。

| 比对结果 | 处理 |
|---------|------|
| **推导 ≈ 账单**（容差 ≤¥0.02） | 通过 |
| **推导 ≠ 账单** | **脏值或公式缺项**。AI 必须：① 检查是否有 **新增费用类型 / 新列** 在账单中出现但当前公式枚举未覆盖；② **主动通知 xiaohou**，说明差异金额、涉及的费用类型/列名、建议归入哪个分项；③ **确认后**更新公式枚举，**不得**自行默认跳过或静默补零 |

**禁止**的做法：
- 发现不等后不告知、静默按账单值替代推导值
- 凭主观猜测某类型「应该归入 XX 分项」后直接改规则（**必须先通知 xiaohou**）
- 用订单侧字段估算替代财务账单实际值
- 订单匹配不到财务账单时默默跳过不列出

**脚本层面**：可选列（**`阶梯扣点佣金`**、**`包装费`**）表头**存在**则加入求和；**不存在**时跳过并 stderr 提示。**无论是否跳过，最终推导与应结的差异才是判定标准**——差异 >0.02 就是 FAIL，脚本输出失败明细。同时输出未匹配订单汇总。

## 3. 应结金额恒等式（已用样本 8 单验证）

各金额列按 Excel **已带正负号** 的数值 **直接相加**（勿再对佣金、基础服务费人为翻号）。

```
应结金额
  = 订单原价
  + 平台承担补贴(市场费)
  + 商家承担货款补贴
  + 商家承担运费补贴
  + 商家自送配送费
  + 商家承担小费
  + 取件服务费(开票)(正向单展示远距离运费;售后单则展示达达售后运费)
  + 总佣金(货款佣金+运费佣金+餐盒费佣金)(可开票)
  + 基础服务费
  + 阶梯扣点佣金        （表头有「阶梯扣点佣金」列时）
  + 包装费              （表头有「包装费」列时）
```

**不包含**：**`开票金额`**（任何情况下均勿加入上式）。

**京东小时达（长表）**：对账净额仍为 **`SUM(结算金额)`**。经营分项时，凡 **`费用类型`** 为 **`到家包装费`** 或与平台导出字面一致的 **`包装费`**（以账单为准），**单独归入「包装费」分项** **`SUM(结算金额)`**；**不得**再计入 **「商家补贴」** 分项。**账单未出该类型行则当期「包装费」分项不参与**（与 §1.1 一致）。

**概念对应（与你口述对齐）**：

- **货款** = `订单原价`  
- **活动补贴** = `平台承担补贴(市场费)`  
- **财务运费等相关支出（样本中可拆列求和）** = `商家承担货款补贴` + `商家承担运费补贴` + `商家自送配送费` + `商家承担小费` + `取件服务费(开票)…`  
- **包装费** = **到家**：仅宽表 **`包装费`** 列（§3 公式）；**小时达**：仅包装类 **`费用类型`** 行单独成项（**非**商家补贴）  
- **平台扣点佣金** = `总佣金(…)(可开票)` + `基础服务费` + **`阶梯扣点佣金`**  

**容差**：单条 **≤ ¥0.02** 视为通过（解析/浮点）。

**本样本结果**：`商家对账单_20260329.xlsx` 中 **`阶梯扣点佣金`**、**`包装费`** 均为 0，纳入公式后仍为 8/8 单 `|应结金额 − 推导|` = 0；`第1页` **`SUM(应结金额)`** = **`第2页` `合计应结金额`** = **655.71**。

## 4. 与数仓的关系

- 当前 `ods_rpa_jd_finance` 为 **长表**（`费用类型` + `结算金额`），**不含** 上述宽表列；**到家** 订单与现有财务长表 **号段/入仓范围** 可能不一致。  
- **入仓后**：应按本文件列名建表或列映射，再对 `订单号` 跑 §3 恒等式。  
- **例行校验脚本**：仓库内 **`scripts/validate_jd_daojia_merchant_statement.py`**，传入本类 xlsx 路径即可复验。
