# 饿了么：结算金额字段对照（财务账单 vs 订单导出）

> **硬性口径（对账）**：与订单导出 **商户应收** 对齐的是财务账单 **`SUM("订单应收")`**（= 11 项公式还原），**不是**「净营业额合计」。  
> **完整「结算金额列」**：平台 `"结算金额"` 按行汇总时，**赔偿类行**（`"订单类型"` 含「赔偿」）上 **`"订单应收"` 为 0**，仅 **`"结算金额"` 有值**，须 **订单应收汇总 + 赔偿行结算金额** 才等于 **`SUM("结算金额")`**。

## 1. 主口径（以财务账单为准）

| 业务含义 | 财务账单（`ods_rpa_eleme_fin_sales_detail`） | 订单导出（`ods_rpa_eleme_order`） | 说明 |
|----------|-----------------------------------------------|-----------------------------------|------|
| **订单应收（对账主口径）** | 窗内同一订单号 **`SUM("订单应收")`**（可多行轧差） | **`商户应收金额`** | 与订单导出、11 项公式还原一致；**禁止**只取 `DISTINCT ON` 一行代替汇总。 |
| 净营业额（参考） | `商家收入_净营业额合计_净营业额合计` | — | **不等于**对账口径；仅作分析参考，**禁止**当作 `o2o_daily_store_aggregate` 的 `ele_settlement_amt`。 |
| **结算金额列（平台列）** | **`SUM("结算金额")`** | — | **非赔偿行**：与该行 `"订单应收"` 一致；**赔偿行**（`"订单类型"` 含「赔偿」）：`订单应收`=0，**仅结算列有值**。故 **`SUM("结算金额") = SUM("订单应收") + SUM("结算金额" \| 赔偿类)`**。例 3.25 匹配单：26420.01 = 26396.36 + 23.65（赔偿单 8.78+14.87）。 |

### 1.0 赔偿单与结算列（2026-03-28 验证）

- 判断：**`"订单类型"` LIKE '%赔偿%'**（如「赔偿单」）。
- 现象：该行 **`"订单应收"` = 0**，**`"结算金额"` > 0**（或负值，按平台实际）。
- 与对账关系：**订单侧 / 公式还原 / 订单应收** 仍按 **非赔偿行 + 赔偿行订单应收（为 0）** 汇总，与 **`SUM(订单应收)`** 一致；若要与 **`SUM("结算金额")`** 对齐，需 **加上赔偿行的 `"结算金额"` 合计**。

## 1.1 退款 / 取消（**经营对账规则**，已纳入 SQL）

- 条件：**`订单状态 = 已取消`** 或 **`退款状态 = 全部退款`**。  
- 读导出 **「商户应收金额」** 解析值 `raw`：  
  - **`raw > 0`**：对账侧 **按 0**（视为不应再记正向应收）。  
  - **`raw < 0`**：**保留负值**（视为成本支出 / 轧差后净应付等，与财务多行轧差对齐）。  
  - **`raw` 为 NULL 或 0**：对账侧 **0**。  
- 非取消且非全退：对账侧 = 导出解析值（同前）。  
- **财务侧**：窗内 **`SUM("订单应收")`** 按订单号（一单多行须相加）。  
- 主对比 = **规则口径订单侧** vs **财务汇总**；另保留 **诊断**：导出原始 vs 财务（`cnt_orders_诊断_导出原始_ne_财务汇总`）。

## 2. 结算金额还原公式（2026-03-28 最终验证通过，3.25 全量 1370 笔 0 差异）

**行级恒等式**（所有项直接相加，DB 值已带正负号：支出为负、收入为正）：

```
结算金额 = 商品金额 + 应收运费 + 实收佣金 + 实收支付服务费
         + 活动补贴 + 代金券补贴 + 商家配送费补贴总额
         + 履约技术服务费合计 + 物流配送费合计 + 骑手小费 + 优惠金额合计
```

### 易错列对照

| 容易混淆的列 | 实际含义 | 能否入公式 |
|-------------|---------|-----------|
| `平台交易技术服务费` | = 实收佣金 + 实收支付服务费 **汇总** | **不能**与子项同时入公式 |
| `（计费基数子列）` / `（计费基数，重灌后为平台交易技术服务费_6 等子列）` | 佣金/支付费的 **计费基数**（=用户支付金额） | **不能**入公式 |
| `col_57` / `col_61` | 佣金费率 / 支付费率 | 仅供验算 |
| `col_19`（净营业额下） | 类似运费列但**非**应收运费 | 应收运费取 `物流净营业额` |
| `商家收入资金来源_其中：平台承担` | 资金来源维度的活动补贴 | **不能**入公式（会导致双重计入，活动补贴取 `col_20`） |
| `"商家收入_净营业额合计_商品净营业额_5"` | 资金来源维度的代金券 | **不能**入公式（代金券取 `col_21`） |
| `（重灌后不存在）` / `col_75` | 120天全0，不含实际数据 | **不能**入公式（骑手小费取 `"其他支出"`，优惠取 `"其他支出"`） |

## 3. 取数规则（与 SQL 一致）

- 财务：`dt >= 业务日 D - 120` 窗内，按 **`"订单号"` GROUP BY**，对 **`"订单应收"`** 做 **SUM**（每行先正则合法再转 `numeric`，非法当 0）。同一账单日多行、或跨日冲补，均以 **订单维度合计** 对齐订单导出。
- 订单锚点（对比脚本）：可与导出同日全量一致；**门店日聚合** `ele_anchor` 仍可能剔除取消/全退（见 `o2o_daily_store_aggregate.sql`）。

## 4. 相关脚本

- **单笔验证（11项还原）**：`sql/o2o_eleme_fin_single_order_rebuild_verify.sql` — 行级展开 11 项分项 + 公式还原结算金额 vs 实际结算金额；含整单汇总验证。
- **门店日聚合**：`sql/o2o_daily_store_aggregate.sql` → `ele_fin_global.settlement_amt` = **`SUM("订单应收")`** 按订单号。
- **分项 vs 财务对比**：`sql/o2o_eleme_order_fin_settlement_compare.sql` → 逻辑 A = 财务 **订单应收**；逻辑 B = 11 项公式还原；赔偿单补全 `"结算金额"`；与对比脚本 **同一 `params.d`**。
- **按订单类型诊断**：`sql/o2o_eleme_fin_by_order_type_diag.sql` → 按 **`"订单类型"`** 汇总 **`SUM("结算金额") − SUM("订单应收")`**，排查除「赔偿」外是否还有类型行级轧差。
- **三平台粗验**：`sql/o2o_settlement_two_logic_validation.sql` → 饿了么逻辑一 = **订单应收** 汇总。
